## 5. QA Risk Assessment

### Risk 1 — Race condition on concurrent duplicate submissions (HIGH)

**What:** Two clients submit the same `eventId` simultaneously. Both pass `db.query(...).first() == None` before either writes. The second `db.add()` raises a `UNIQUE constraint failed` SQLAlchemy error — currently unhandled, surfaces as HTTP 500.

**Impact:** Financial event is double-processed in the Account Service before the Gateway catches the DB error. Balance is wrong.

**Mitigation:** Wrap the insert in a `try/except IntegrityError` and re-fetch the existing event to return as a duplicate (200).

---

### Risk 2 — Float arithmetic for balance calculation (MEDIUM)

**What:** SQLite stores `amount` as `REAL` (IEEE 754 double). Summing many small floats accumulates rounding error. Example: 100 × 0.10 ≠ 10.00 in floating point.

**Impact:** Reported balance differs from the true financial balance by small fractions — unacceptable for a ledger.

**Mitigation:** Store amounts as `INTEGER` cents (multiply by 100 on input, divide on output) or use Python's `Decimal` type with `NUMERIC` SQLite affinity.

---

### Risk 3 — No currency consistency enforcement (MEDIUM)

**What:** The Account Service accepts `CREDIT 100 USD` and `DEBIT 50 EUR` on the same account. The balance is computed as `50` with no currency attached to the subtraction.

**Impact:** The `/balance` endpoint returns a balance in the currency of the last transaction, not necessarily the account's currency. Balance figure is meaningless.

**Mitigation:** On first transaction for an account, record the account currency. On subsequent transactions, reject mismatched currencies with `422`.
